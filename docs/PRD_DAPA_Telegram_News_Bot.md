# PRD: 방위사업 조간 뉴스 텔레그램 봇

## 1. 개요

방위사업청, 국방부, 국방일보, 정책브리핑, 방산 관련 보도자료와 신뢰도 높은 뉴스 RSS를 매일 아침 07:00(KST)에 수집·정리해 텔레그램으로 발송하는 조간 브리핑 봇을 구축한다.

이 봇의 목적은 직원들이 출근길에 방위사업, 무기체계, 방산수출, 국방정책 동향을 빠르게 파악하도록 돕는 것이다. 국내외 출장 중이거나 내부망/내부 시스템에 접근하기 어려운 직원도 공개 출처 기반 동향을 확인할 수 있어야 한다.

## 2. 문제 정의

현재 방위사업 관련 정보는 여러 공식 사이트, 언론사, 보도자료 배포망에 흩어져 있다. 직원이 매일 아침 직접 확인하기에는 시간이 많이 들고, 출장 중에는 내부 시스템 접근이 어려워 정보 공백이 생긴다.

필요한 것은 내부자료를 외부로 전송하는 시스템이 아니라, 공개 출처의 공식성 있는 자료를 자동 수집해 읽기 쉬운 조간 형식으로 재구성하는 시스템이다.

## 3. 목표

- 매일 07:00(KST)에 방위사업 관련 핵심 동향을 텔레그램으로 자동 발송한다.
- 공식 출처를 우선하고, 보조 뉴스/RSS는 키워드 필터와 중복 제거를 거쳐 사용한다.
- 기사 제목, 출처, 발행일, 링크, 40~80자 요약을 제공한다.
- 섹션별로 방위사업청·국방정책, 무기체계·전력화, 방산수출·방산업계 동향을 구분한다.
- 출장 중인 직원도 모바일 텔레그램에서 빠르게 확인할 수 있게 한다.

## 4. 비목표

- 비공개 내부자료, 내부망 문서, 대외비/군사기밀 정보를 수집하거나 전송하지 않는다.
- 기사 전문을 무단 복제하지 않는다.
- 투자 목적의 방산주/테마주 알림 서비스를 만들지 않는다.
- 텔레그램을 내부 결재, 지시, 업무보고 시스템으로 사용하지 않는다.

## 5. 주요 사용자

- 방위사업, 국방정책, 무기체계, 방산수출 동향을 매일 확인해야 하는 직원
- 국내외 출장 중 내부 시스템 접근이 제한되는 직원
- 아침 회의 전 핵심 이슈를 빠르게 파악해야 하는 실무자/관리자

## 6. 핵심 사용 시나리오

1. 매일 07:00(KST)에 봇이 공식 RSS와 보조 RSS를 조회한다.
2. 최근 24~72시간 기사 중 방위사업 관련성이 있는 항목만 남긴다.
3. 중복 기사와 주식/테마주성 기사를 제거한다.
4. LLM이 섹션별로 기사를 분류하고 짧게 요약한다.
5. 텔레그램 채널 또는 단체방에 조간일보 형식으로 발송한다.
6. 사용자는 출근길 또는 출장 중 모바일에서 핵심 동향과 원문 링크를 확인한다.

## 10. 기능 요구사항

### FR-1. RSS 수집

- 등록된 RSS 피드를 매일 07:00(KST)에 조회한다.
- 각 항목에서 제목, 링크, 발행일, 출처, 요약/설명을 추출한다.
- RSS 응답 실패 시 해당 피드만 실패 처리하고 전체 작업은 계속 진행한다.

### FR-2. 날짜 필터

- 기본 수집 기간은 최근 24시간이다.
- 월요일 또는 휴일 다음날에는 최근 72시간까지 확장할 수 있다.
- 기사 수가 부족하면 3일, 최대 5일까지 재시도한다.

### FR-3. 중복 제거

- 동일 URL은 1건만 남긴다.
- 제목이 거의 같은 기사는 공식 출처를 우선한다.
- 같은 이슈가 여러 언론에 반복 보도된 경우 정책브리핑/국방일보/원보도자료 순으로 우선한다.

### FR-4. 관련성 판정

- 포함 키워드가 제목 또는 설명에 포함된 기사를 우선한다.
- 제외 키워드가 포함된 주식/투자성 기사는 제외한다.
- 애매한 기사는 LLM 분류 단계에서 제외할 수 있다.

### FR-5. LLM 요약

- 기사 전문을 수집하지 않는 구조를 기본으로 한다.
- 제목, 출처, 발행일, RSS 설명, URL만 기반으로 요약한다.
- 요약은 40~80자 내외로 작성한다.
- 불확실한 내용은 단정하지 않는다.

### FR-6. 텔레그램 발송

- `pyTelegramBotAPI`의 `telebot.TeleBot`을 사용한다.
- 정기 발송에는 polling이 필요하지 않으며, `send_message(chat_id, text, parse_mode="HTML")` 형태로 전송한다.
- 메시지가 Telegram 길이 제한을 넘으면 섹션 단위로 분할 발송한다.
- 채널 발송 시 봇을 채널 관리자로 추가해야 한다.

### FR-7. 운영 알림

- 수집 결과가 0건이면 "오늘 수집된 공개 방위사업 관련 기사가 없습니다"를 발송한다.
- 발송 실패 시 로그를 남기고 1~2회 재시도한다.
- 매일 발송 결과를 로컬 로그 또는 GitHub Actions 로그에 남긴다.

## 11. 비기능 요구사항

- 보안: 봇 토큰, LLM API 키, 텔레그램 chat_id는 `.env` 또는 GitHub Secrets에 저장한다.
- 개인정보: 수신자 개인 전화번호나 계정 정보를 저장하지 않는다.
- 기밀성: 공개 RSS와 공개 뉴스만 사용한다. 내부자료는 입력하지 않는다.
- 안정성: 일부 RSS가 실패해도 나머지 피드로 브리핑을 생성한다.
- 재현성: RSS URL, 키워드, 프롬프트, 실행 시간을 코드/설정으로 관리한다.
- 확장성: 향후 Slack, 이메일, Notion, 사내 포털로 발송 채널을 추가할 수 있게 모듈화한다.

## 12. 권장 기술스택

### 언어/런타임

- Python 3.11 이상

### RSS/HTTP

- `feedparser`: RSS/XML 파싱
- `httpx`: HTTP 요청, timeout, retry 처리

### 텔레그램

- `pyTelegramBotAPI`: `telebot` 기반 텔레그램 발송
- Telegram Bot API: BotFather로 토큰 발급, `sendMessage` API 사용

### 스케줄링

- 1순위: Linux 서버/VPS의 `cron` 또는 `systemd timer`
- 2순위: Windows 작업 스케줄러
- 3순위: GitHub Actions scheduled workflow

주의: GitHub Actions cron은 UTC 기준이며 실행 지연이 있을 수 있다. 07:00 KST는 UTC 22:00(전날)이므로 `0 22 * * *`로 설정한다. 정확한 발송 시간이 중요하면 항상 켜져 있는 서버의 cron을 권장한다.

### LLM 요약

- OpenAI API 또는 사내 허용 LLM API
- 모델명은 코드에 하드코딩하지 않고 `LLM_MODEL` 환경변수로 관리한다.

### 설정/비밀값

- 로컬: `.env`
- GitHub Actions: Repository Secrets
- 필수 환경변수:

```text
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
LLM_API_KEY=
LLM_MODEL=
TZ=Asia/Seoul
```

## 15. LLM 프롬프트

### 15.1 시스템 프롬프트

```text
당신은 방위사업, 국방정책, 무기체계, 방산수출 동향을 정리하는 조간 브리핑 편집자다.

입력으로 제공되는 정보는 공개 RSS와 공개 뉴스의 제목, 출처, 발행일, URL, 짧은 설명뿐이다.
기사 전문에 없는 사실을 만들지 않는다.
군사기밀, 내부정보, 추정성 작전정보를 생성하지 않는다.
투자 조언이나 주식 매매 판단을 제공하지 않는다.

목표는 직원들이 출근길 또는 출장 중 모바일에서 방위사업 관련 공개 동향을 빠르게 파악하도록 돕는 것이다.
공식 출처와 정부 발표를 우선하고, 보조 언론/보도자료는 관련성이 높은 경우에만 포함한다.

출력은 한국어로 작성한다.
각 기사 요약은 40~80자 내외로 간결하게 작성한다.
불확실한 내용은 단정하지 말고 "발표했다", "보도했다", "전했다"처럼 출처 기반 표현을 사용한다.
```

### 15.2 사용자 프롬프트 템플릿

```text
아래 기사 후보를 검토해 방위사업 조간 브리핑을 작성하라.

[작성 기준]
1. 최근성, 공식성, 방위사업 관련성을 기준으로 우선순위를 정한다.
2. 아래 3개 섹션으로 분류한다.
   - 방위사업청·국방정책
   - 무기체계·전력화
   - 방산수출·방산업계
3. 각 섹션은 최대 3건까지 선정한다.
4. 중복 이슈는 공식 출처 또는 원출처 1건만 남긴다.
5. 방산주, 수혜주, 목표가, 증시성 기사는 제외한다.
6. 기사 전문에 없는 내용은 추측하지 않는다.
7. 각 요약은 40~80자 내외로 작성한다.
8. 링크는 반드시 입력 URL을 그대로 사용한다.
```

### 15.3 관련성 판정 프롬프트

```text
다음 기사 후보가 방위사업 조간 브리핑에 포함될 만한지 판정하라.

판정값:
- include: 포함
- exclude: 제외
- maybe: 애매함

포함 기준:
- 방위사업청, 국방부, 국방획득, 방위력개선, 국방조달, 무기체계, 전력화, 방산수출, K-방산과 직접 관련

제외 기준:
- 방산주, 테마주, 수혜주, 목표가, 증시, 단순 기업 주가 기사
```

## 16. 텔레그램 연동 설계

### 16.1 BotFather 준비

1. Telegram에서 `@BotFather`를 연다.
2. `/newbot`으로 봇을 생성한다.
3. 발급된 bot token을 `TELEGRAM_BOT_TOKEN`에 저장한다.
4. 발송 대상이 채널이면 봇을 채널 관리자로 추가한다.
5. 발송 대상이 단체방이면 봇을 초대한 뒤 chat_id를 확인한다.

### 16.2 발송 코드 개념

```python
import os
import telebot

bot = telebot.TeleBot(os.environ["TELEGRAM_BOT_TOKEN"], parse_mode="HTML")
bot.send_message(
    chat_id=os.environ["TELEGRAM_CHAT_ID"],
    text=message,
    disable_web_page_preview=False,
)
```

정기 발송만 필요하면 `bot.infinity_polling()`은 사용하지 않는다. `/start`, `/latest`, `/help` 같은 대화형 명령을 지원할 때만 polling 또는 webhook을 추가한다.

## 17. 스케줄링 설계

### 17.1 서버 cron 권장

```cron
0 7 * * * cd /opt/news-dapa && /usr/bin/python -m dapa_news_bot.run_daily
```

서버 timezone은 `Asia/Seoul`로 설정한다.

### 17.2 GitHub Actions 대안

```yaml
on:
  schedule:
    - cron: "0 22 * * *"  # 07:00 KST
  workflow_dispatch:
```

GitHub Actions는 UTC 기준이며 실행 지연이 있을 수 있으므로 엄밀한 07:00 발송이 필요하면 서버 cron을 사용한다.

## 19. 성공 지표

- 평일 07:00~07:05(KST) 내 발송 성공률 95% 이상
- 공식/준공식 출처 비중 70% 이상
- 중복 기사 포함률 10% 이하
- 주식/테마주성 기사 포함률 5% 이하
- 브리핑 1회당 읽는 시간 3분 이하

## 20. 운영 리스크와 대응

| 리스크 | 대응 |
|---|---|
| RSS 주소 변경 | 피드별 실패 로그와 주 1회 점검 |
| 기사 부족 | 기간을 3~5일로 확장하고 부족 사실 명시 |
| Telegram 토큰 유출 | `.env`/Secrets 사용, 유출 시 BotFather에서 토큰 재발급 |
| 비공개 정보 혼입 | 공개 RSS만 입력으로 사용, 내부자료 업로드 금지 |
| LLM 환각 | 제목/출처/설명 기반 요약만 허용, 원문에 없는 내용 금지 |
| GitHub Actions 지연 | 정확한 발송은 서버 cron으로 운영 |

## 21. MVP 범위

1. RSS 피드 설정 파일 작성
2. RSS 수집 및 표준 JSON 변환
3. 키워드 포함/제외 필터
4. 중복 제거
5. LLM 요약/분류
6. Telegram HTML 메시지 생성
7. 07:00(KST) 스케줄 발송
8. 실행 로그 저장

## 22. 향후 확장
